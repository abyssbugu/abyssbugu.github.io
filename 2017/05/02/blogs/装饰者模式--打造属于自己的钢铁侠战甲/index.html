装饰者模式-打造属于自己的钢铁侠战甲

发表于 2017-05-02 | 分类于 设计模式

背景介绍

装饰者模式也称为包装模式,属于结构型设计模式,本次通过学习装饰者模式,运用对装饰者模式的理解,打造一套属于自己的钢铁侠战甲!让我们开始吧~

定义

Decorator Pattern，是在不必改变原类文件和使用继承的情况下，动态地扩展一个对象的功能,它是通过创建一个包装对象，也就是装饰来包裹真实的对象。
装饰者是代替继承扩展功能另外一种方式，装饰者模式遵循了多用组合少用继承和对扩展开放对修改关闭的设计原则。

UML类图

  • Component(抽象构件):定义一个抽象接口,规范待装饰者的功能和方法
  • ConcreteComponent(具体构件):实现抽象接口的待装饰类
  • Decorator(装饰者):持有Component对象的实例,并定义一个与抽象构件接口一致的接口
  • ConcreateDecorator(具体装饰):是Decorator的子类也是Component的子类,在持有Component的引用后，由于其自身也是Component的子类，那么，相当于ConcreteDecorator包裹了Component，不但有Component的特性，同时自身也可以有别的特性，也就是所谓的装饰。

拓展

关于Android源码中装饰者的使用

  • Context - 抽象组件

  • ContextImpl - 具体组件:继承自Context,实现了Context中的抽象方法;

  • ContextWrapper - 装饰者抽象:继承Context,定义了一些增强方法,内部持有Context引用

  • ContextThemeWrapper - 具体装饰者父类:实现了ContextWrapper中的抽象方法,提供一些通用实现

  • Activity,Service,Application - 具体装饰者实现类:实现全部的装饰方法

特点

  • 装饰者和被装饰者有相同的接口（或有相同的父类）。
  • 装饰者保存了一个被装饰者的引用。
  • 在运行时动态地为对象添加属性，不必改变对象的结构。
    使用装饰者模式的最大好处就是其拓展性十分良好，通过使用不同的装饰类来使得对象具有多种多样的属性，灵活性比直接继承好。然而它也有缺点，那就是会出现很多小类，即装饰类，使程序变得复杂。
